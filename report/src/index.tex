\subsection{Abstract}
Many stores use point of sale (POS) systems to manage their inventory and sales.
These systems help store owners track sales and manage their inventory. These
are often touch-operated devices that also give the option of using a mouse
pointer or keyboard shortcuts. They consist of purpose-built machines running
proprietary, often custom-built, software.

Due to this proprietary nature, these systems are often expensive and are costly
and time consuming to update. This incurs higher maintenence costs on businesses
ityand a higher workload on employees due to the unreliabile of outdated software and hardware.

\subsection{Project Stakeholders}
Potential project stakeholders would include any physical retailer with the
desire to use this software. The goal is to design the software such that it can
be adapted to be used in any setting, offering a decent amount of customization. Although, the main
target audience, in the beginning, will be traditional storefront retailers.

\subsubsection{Overview}
The system that is created is one which accomodates the following functions:
\begin{itemize}
  \item Inventory Management
  \item Payment Management
  \item Transaction Management
\end{itemize}

following these requirements, the system will take on a Client-Server Archetecture, with
the backend acting as a server, and the multiple POS terminals acting as its clients.\par
This approach has many benefits, such as:


\begin{enumerate}
\item Safety. If multiple terminals all need to keep track of their own databases, in
seperate instances, it increases the difficulty and risk of miscounting and other data errors.
When numbers are centralized and updated in a queue or other safe data structure, this
risk decreases significantly.
\item Flexibility. Any number of these terminals can be added with no downside.
\item Simplicity. A POS terminal does not need specialized hardware and software, only a network
connection, and a modern internet browser. Updates can also be handled over the air and updates 
to the frontend can be serverside.
\end{enumerate}

To describe an example of the implementation, you could have a server, say a desktop running
mongoDB through Rust, communicating with many terminals, which could be running android or some
other lightweight linux, locked down into a browser environment, which holds the frontend app, 
which is programmed in Svelte script.\par
This portion of the document will be updated with further details as the project is built.

\subsubsection{Subsystem Decomposition}

The major subsystems of the Client-Server approach are the user layer, application layer,
and the data layer.\par
The user layer encompases the user-iinterface and all mechanical systems that the user interacts with
such as the physical cash register or touch screen. This portion of the system is
ornamental, but also serves the important role of being the way in which the system
is experienced. Our team is using Svelte to create the user interface, which is the only part of the user layer 
which is part of the app. Svelte can create complex user interface quickly and easily,
which makdes it ideal for this usage case.\par
Tha application layer is the portion of the app that does the ``thinking'' so to speak.
While exact datastructures and implementations have not been finalized, the idea is to have
a list of items which can be edited from within the frontend, until that list is finalized and sent
over the network to the database as a list of required updates to make. This process
happens before payment in attempt to prevent miscounts.\par
The data layer is contained almost entierly within the backend. It stores inventory along with the
UPCs (Universal Product Code) and the price for each item. This portion of the software will be done
using MongoDB through Rust.

\subsection{User Interface Design}

The first screen the user will see is the login screen. Here the user can log in to their
account; no other pages can be accessed until the user is logged in.\par
Once the user is logged in, they are presented with the main screen. The main screen is the checkout
screen, where items can be scanned in to be included in the transaction.\par
At the top of the screen, there is a bar containing the log out button, and 
if the logged-in user has admin privilages, there will also be an option to
go to the admin settings page.\par
The bulk of the main page shows the items that are to be purchased including name, quantity,
and price. At the bottom of this view, there will be a line showing the total price and quantity.\par
At the bottom of the main page there is a checkout button.
When the ``checkout'' button is presseed, the program enters the checkout screen. Here, a total
will be displayed, along with tax and other details. There will be buttons for the user to pick between cash or card.
Pushing ``cash''s will prompt the user to enter the cash recieved, and display what
change to give. Pressing ``check'' will turn control of the program to the card reader
to process the payment, and wait until the payment has finished processing.\par
After one of these buttons is pressed and the transaction is complete, the program will return to the
main screen.

\subsection{Limitations and
Recommendations}

% Include a brief description of the limitations of your current prototype
% that you are aware of, and discuss possible solution approaches to
% overcome those limitations.

% Please keep the discussion brief. 1 page text should be sufficient for

% Note that there might be overlap between the Limitations and

The biggest limitation of our software is that it is not able to process card transactions.
As it is set up right now, the system will work fine for a cash only workflow, but an API to process
cards is missing.\par
Our system also lacks the nessecary interfaces and general programming to accept device peripherals
one would expect to be able to hook up to a POS system, most notably, a barcode reader. As of right now,
all barcodes must be implemented by hand, which is not ideal.

although this was very close to happening, it was unfortunetly not able to be implemented

Our team generally had two weekly meetings. On Tuesdays meeting at 1:30pm to 3:00pm
to discuss general plans catch each other up on progress and also serving as just a
reserved time to work on the project. On Thursdays, we would meet with the professor
for 15 minutes to discuss upcomming plans for the next week, and to stay informed on
upcomming events in the class. Thursdays would also typically hold impromptu meetings
1 hour to 45 minutes before the meeting with the professor to catch up, and to make
sure all our work merges correctly.\par

Group communication primarilly occured through discord, although these mainly served
as reminders due to the extended meeting times. Occasionally GitHub would be utilized
as a communication tool through push messages and times, although this mainly served
to alert people if one person was actively working.
